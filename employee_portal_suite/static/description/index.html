Module Overview

Employee Portal Suite is a comprehensive Odoo module that creates a complete self-service environment accessible at /my/employee.

What the module provides

📋
Employee Requests
Multi-level approval workflow

Employees submit requests (leave, salary advance, training, etc.) that route through Manager → HR → Finance → CEO. Each stage sends notifications and logs approval metadata.

📦
Material Requests
6-stage procurement workflow

Employees request materials linked to their project. Requests flow through Purchase Rep → Store Manager → Project Manager → Director → CEO with quotation tracking and PO linking.

📍
GPS Attendance
Geofenced check-in / check-out

Employees check in/out from the portal. Each work location has configurable GPS coordinates and an allowed radius. Out-of-range checkouts are flagged and HR is notified.

✍️
Document Signing
Integrated with Odoo Sign

Pending signature requests from Odoo Sign surface in the employee portal dashboard. Employees can track and complete signatures without leaving the portal.

How to assign groups in Odoo

Navigate to Settings → Users
Go to Settings → Users & Companies → Users. Open the target user's form.
Find the Employee Portal section
Scroll to the Employee Portal group category that the module adds. You'll see individual group checkboxes for each role.
Also ensure the user is a Portal user
For employees who use the portal (not the backend), their user type must be set to Portal. Backend users (HR, Finance, Admin) use Internal User type.
Link the user to an hr.employee record
Go to Employees → find or create the employee → in the HR Settings tab, set the Related User. Without this link, the portal redirect will not work and forms will raise an error.
⚠️ A user assigned the group_attendance_only group will always be redirected directly to /my/employee/attendance — they cannot navigate to any other section of the portal. Assign this group only to workers who exclusively need clock-in/out access.

Record isolation rules

Three record-level rules (ir.rule) are defined to ensure data privacy:

1. Employee — own requests only
domain_force: [('employee_id.user_id', '=', user.id)]
Applied to both employee.request and material.request for the Employee group.
2. Manager — own + subordinates
domain_force: ['|', ('manager_id.user_id', '=', user.id), ('employee_id.user_id', '=', user.id)]
Managers see requests where they are the direct manager OR where they are the requester. Applied to employee.request only.
ℹ️ The Manager record rule relies on manager_id being correctly populated. This field is auto-computed from the employee's parent_id in the HR module. Make sure each employee has a "Manager" set in their employee record.

Work Locations & GPS Configuration

Each work location must be configured with GPS coordinates for geofencing to work. This controls where employees can check in and out from.

Fields on hr.work.location

Field Type Description
project_id Many2one Required. Links this location to a project for Material Request routing.
geo_latitude Float(10,7) GPS latitude of the work location center (e.g. 24.7136 for Riyadh).
geo_longitude Float(10,7) GPS longitude of the work location center (e.g. 46.6753 for Riyadh).
geo_radius Integer Allowed radius in metres (default: 200). Set to 0 to disable geofencing.
geo_enforce Boolean Master switch. If False, employees can check in from anywhere regardless of radius.

How geofencing works

📍
👷
👷
✓ Within radius
⚠ Outside radius
radius = 200m

The system uses the Haversine formula to calculate the great-circle distance between the employee's reported GPS position and the work location center. If the distance exceeds geo_radius AND geo_enforce = True, the check-in is blocked. Check-outs outside the radius are allowed but flagged with checkout_outside_location = True and HR is notified via an automated action.

Configuration steps

Open the Work Location form
Go to Employees → Configuration → Work Locations. Create or edit a work location.
Set the linked Project
The project_id field is required. This links the location to the project for material request routing. Set it to the relevant project.
Enter GPS coordinates
Enter geo_latitude and geo_longitude. Use Google Maps — right-click on the site location and copy the coordinates. E.g. 24.7136, 46.6753.
Set the allowed radius
Enter geo_radius in metres. For an office, 50–100m is typical. For a construction site, 200–500m may be appropriate depending on site size.
Enable enforcement
Check Enforce Geofencing. Without this, coordinates and radius are saved but not enforced — useful for setting up before going live.
Assign to employees
On each employee record, set their Work Location to this location. The attendance portal will then use this location's GPS settings for their check-ins.
✅ If geo_latitude and geo_longitude are both 0 (or not set), geofencing is bypassed even if geo_enforce = True. This prevents accidental lockouts during setup.

Projects Configuration

Projects are central to Material Request routing. Each project must have a Store Manager and Project Manager assigned for the approval chain to work correctly.

Fields added to project.project

FieldDescriptionImpact
store_manager_employee_id The employee who acts as Store Manager for this project Material requests in store state are routed to this person's linked user for approval
project_manager_employee_id The employee who acts as Project Manager for this project Material requests in project_manager state are routed to this person for approval
⚠️ If a project's Store Manager or Project Manager employee does not have a linked Odoo user (employee.user_id), the approval notification will fail silently — no email or activity will be created for that stage. Always verify the employee→user link.

How employee project link works

When a Material Request is created, the system auto-fills the project from the employee's work location:

# Auto-computed on material.request project_id = compute("_compute_project_from_employee") # → reads employee_id.work_location_id.project_id store_manager_user_id = project.store_manager_employee_id.user_id project_manager_user_id = project.project_manager_employee_id.user_id

This chain means: Employee → Work Location → Project → Store Manager / Project Manager. All three links must be intact.

For each stage, the system records:

Quotation tracking

The module tracks how many supplier quotations have been uploaded as attachments. The system requires 3 quotations before the request can proceed in the standard flow.

Admins can override the quotation requirement using Quotation Status Override — set to Exception Approved or Single Source Approved to bypass the 3-quotation rule.

Post-approval features

Purchase Order linking

Once a PO is created referencing the Material Request, a smart button appears on the MR form. The No PO Required checkbox can be set if the purchase is too small to require a PO (e.g. petty cash).

Invoice / accounting submission

After purchase, the portal user can upload accounting documents and submit them to accounting. The status tracks: Pending Invoice → Submitted → Received → Processed (or Rejected / Need More Docs).

Clarification requests

Any approver can request clarification before approving. This pauses the workflow and shows a "Clarification" badge on the request. The requester responds and the approver resumes.

Expense account

Admins and backend users can assign an Expense Account (filtered to expense account types) to the material request for accounting integration.

GPS Attendance

Employees check in and out via the portal at /my/employee/attendance. The module extends Odoo's built-in hr.attendance model with GPS enforcement and auto-checkout.

Check-in / check-out flow

Employee opens the attendance page
The browser requests the employee's GPS location via the Geolocation API. The employee must grant permission.
System validates location against work location
The portal controller calls work_location.check_employee_in_range(lat, lon). If geo_enforce = True and the employee is outside the radius, check-in is blocked with an error message.
Attendance record is created
A new hr.attendance record is created with check_in set to the current UTC timestamp. The portal context key from_portal_checkout=True is set so the auto-checkout flag is NOT triggered.
Check-out: location re-validated
On checkout, location is checked again. If outside radius, checkout is still recorded (not blocked) but checkout_outside_location = True is set on the record and HR is notified via an automated action.

Auto-checkout cron

⏰
Daily auto-checkout at 11:50 PM Riyadh time
Cron: Employee Portal: Auto Check-Out Open Attendances

Runs daily at 20:50 UTC (= 23:50 Asia/Riyadh). Closes all attendance records that are still open (no check_out) from earlier that day.

Why custom? Odoo's built-in auto-checkout skips employees with no work schedule or a flexible schedule. Portal-only employees typically have no schedule and are ignored by the native cron. This custom cron applies to ALL employees.
Detection: Auto-closed records are flagged with auto_checked_out = True and a chatter note is posted. HR staff can filter for these records in the Attendance backend view.

Flags on attendance records

FieldSet whenHR action
checkout_outside_location Employee checks out from outside the allowed GPS radius Automated action posts a chatter note; HR can filter these records
auto_checked_out The daily cron closes an open attendance record Chatter note posted; filter in Attendance list view

Automation & Scheduled Actions

The module installs two automated processes that run without any manual intervention.

1. Daily auto-checkout cron

⏰
Auto Check-Out Open Attendances
Runs every day at 20:50 UTC (23:50 Riyadh time)

Finds all open hr.attendance records (no check_out) from earlier that day and closes them at the moment the cron runs.

To adjust the cron schedule:
Go to Settings → Technical → Automation → Scheduled Actions → find Employee Portal: Auto Check-Out Open Attendances → change the Next Execution Time or interval.

2. Outside-location checkout notification

🚨
Notify HR on Outside-Location Check-out
Triggered when checkout_outside_location flips to True

This is an Odoo Automated Action (base_automation) — not a cron. It fires immediately whenever checkout_outside_location changes from False to True on an attendance record. It calls _notify_hr_outside_checkout() which posts an internal chatter note on the attendance record, visible to HR in the backend.

⚠️ This automated action is active by default. To disable it (e.g. if you don't want HR notifications on out-of-area checkouts), go to Settings → Technical → Automation → Automated Actions → find the action and uncheck Active.
